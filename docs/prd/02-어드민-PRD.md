# 어드민 PRD (Admin Product Requirements Document)

> 목적: GPTers 관리자 대시보드 UX 재설계
> 상태: 초안 (2026-02-25)
> 관련 개선방향: D8 (관리자 대시보드 재설계), D15 연계 (LMS 어드민 기능)
> 해결 대상 문제: P13, P14, P15, P16, P17, P22 + D15 연계 (수료/환급/기수/세션/과제권한)

---

## 1. 개요

### 1.1 목적

Bettermode 기반 관리 패널의 구조적 한계를 해소하고, 운영 효율을 극대화하는 자체 관리자 대시보드를 설계한다. 현재 운영자들은 하나의 작업을 수행하기 위해 7단계 네비게이션을 거치거나, Retool 등 외부 도구 11개에 의존하고 있다. 이를 1~2클릭 접근 가능한 flat 구조 대시보드로 재설계한다.

### 1.2 대상 사용자

| 역할 | 설명 | 주요 작업 | 접근 범위 |
|------|------|----------|----------|
| **운영자 (대표/팀장)** | 신연권 등 커뮤니티/스터디 운영 총괄 | 스터디 상태 관리, 게시글 관리, 배너 수정, 회원 관리 | 전체 |
| **마케터** | 재호 등 콘텐츠/광고 담당 | 배너 수정, 문구 수정, 상품/쿠폰 관리, 프로모션 설정 | 배너, 상품/쿠폰, 텍스트 관리 |
| **스터디장** (제한적 접근) | 각 스터디 리더 | 자기 스터디 상세페이지 수정, 최종제출 토글, 참여자 현황 확인 | 자기 스터디 한정 |

### 1.3 현재 문제 요약

현재 Bettermode 관리 패널 + 외부 도구 의존 구조에서 다음 6가지 핵심 문제가 존재한다:

1. **7단계 네비게이션** (P13) -- 배너 하나 수정에 9단계 클릭
2. **프론트-어드민 매핑 불명확** (P14) -- 어떤 스페이스를 수정하면 어디가 바뀌는지 모름
3. **게시글 이동 불가 + 웹훅 중복** (P15) -- 삭제-재게시 우회로 인한 시스템 오류
4. **Retool 의존** (P16) -- 상품/쿠폰 관리가 외부 도구에 분리
5. **텍스트 수정 제약** (P17) -- 마케터가 문구를 자유롭게 수정 불가
6. **외부 도구 11개 의존** (P22) -- 사이트보다 외부 도구를 더 많이 사용

## 2. 현재 문제 상세 분석

### 2.1 P13: 7단계 네비게이션

**현상**: 홈 배너를 수정하려면 다음 경로를 거친다:

```
관리 → 디자인 스튜디오 → 컬렉션 및 공간 → 컬렉션 없음 → 홈 → Body → 판매주간 홈배너 → Column 2 → 게시물 6
```

**영향**: 운영자(신연권)가 "관리 탭의 UI가 복잡해서 원하는 부분을 찾기 위해 여러 번 헤매곤 한다"고 증언. 자주 쓰는 기능도 깊은 경로에 묻혀 있어 숙련자조차 시행착오를 반복한다.

**목표 상태**: 자주 쓰는 기능(배너, 게시글, 스터디)에 대시보드에서 **1~2클릭**으로 접근.

### 2.2 P14: 프론트-어드민 매핑 불명확

**현상**: 마케터(재호)가 "어떤 스페이스를 수정해야 원하는 UI가 변경되는지 찾기가 어려움"이라고 증언. Bettermode의 스페이스 구조와 실제 프론트 페이지 간의 연결이 불투명하다.

**영향**: 수정 시행착오가 반복되고, 잘못된 영역을 수정하는 실수가 발생한다. 결과적으로 텍스트 하나 바꾸는 데 불필요한 시간이 소요된다.

**목표 상태**: 관리자 UI의 각 설정이 프론트 페이지 어느 영역에 반영되는지 명확히 표시. 미리보기 기능 제공.

### 2.3 P15: 게시글 이동 불가 + 웹훅 중복

**현상**: 회원이 잘못된 게시판에 글을 올리면, 관리자가 해당 게시글을 다른 카테고리로 이동할 수 없다. 유일한 방법은 **삭제 후 재게시**인데, 이 과정에서 n8n 웹훅이 2회 발동되어 자동 요약이 중복 생성된다.

**영향**: 다혜가 "자주 발생하는 문제"로 언급. 중복 데이터 정리에 추가 시간 소요. 회원 입장에서도 글이 삭제되었다가 재게시되는 불쾌한 경험.

**목표 상태**: 관리자가 게시글의 카테고리를 직접 이동 가능. 이동 시 웹훅은 발동하지 않음.

### 2.4 P16: Retool 의존 (상품/쿠폰)

**현상**: 재호가 "상품을 생성하고 쿠폰을 연결하는 과정이 리툴이라는 별도 도구를 사용해야 해서 불편"이라고 증언. 상품 CRUD와 쿠폰 관리가 Bettermode 어드민 밖의 Retool에 분리되어 있다.

**영향**: 워크플로우가 분산되고, Retool 접근 권한/학습 비용이 추가로 발생한다.

**목표 상태**: 어드민 내에서 상품/쿠폰 CRUD 완결. Retool 의존 제거.

### 2.5 P17: 텍스트 수정 제약

**현상**: 재호가 "각 섹션에서 타이틀이나 문구 등을 자유롭게 수정하는 데 제약사항이 많다"고 증언. 배너 문구, 섹션 타이틀, CTA 텍스트 등을 변경하려면 개발자 수준의 접근이 필요하거나 제약이 존재한다.

**영향**: 마케팅 캠페인에 맞춰 빠르게 문구를 교체하는 것이 불가능. 시의성 있는 마케팅 활동에 병목.

**목표 상태**: 마케터가 주요 영역(배너, 섹션 타이틀, CTA 문구)의 텍스트를 직접 수정 가능.

### 2.6 P22: 외부 도구 11개 의존

**현상**: 운영자(신연권)가 사이트보다 외부 도구를 더 많이 사용한다. "사이트 자체가 불편하게 느껴져서 잘 안 쓰려고 한다."

| 외부 도구 | 역할 | 어드민 내재화 가능 여부 |
|-----------|------|----------------------|
| 슬랙 | 팀 소통 | 유지 (팀 소통은 전용 도구가 적합) |
| 에어테이블 | 운영 DB, 스터디 데이터 | 부분 내재화 (스터디/수강생 관리) |
| n8n | 자동화 (요약, 권한 부여) | 내재화 (백엔드 자동화) |
| 노션 | 문서 정리 | 유지 (문서 도구는 별도가 적합) |
| 구글시트 | 표 형태 데이터, UTM 생성 | 부분 내재화 (대시보드 통계) |
| 리니어 | 프로젝트 관리 | 유지 (개발 도구) |
| Retool | 상품/쿠폰 관리 | **완전 내재화** (핵심 대상) |
| 지메일 | 이메일 | 유지 |
| 구글 캘린더 | 일정 관리 | 유지 |
| Zoom | 화상회의 | 유지 (링크 관리만 내재화) |
| 카카오톡 | 팀/수강생 메시지 | 유지 |

**내재화 대상**: Retool(상품/쿠폰), 에어테이블 일부(스터디 데이터 조회), n8n(자동화 로직 백엔드화)

## 3. 핵심 기능 정의

### 3.1 대시보드 (`/admin`)

운영자가 로그인 후 가장 먼저 보는 화면. 오늘의 핵심 지표와 자주 쓰는 작업에 즉시 접근 가능.

#### 3.1.1 오늘의 요약 (통계 카드)

| 지표 | 설명 | 데이터 소스 |
|------|------|------------|
| 신규 가입 | 오늘 가입한 회원 수 | Supabase auth |
| 게시글 작성 | 오늘 작성된 게시글 수 (피드 포스트 + 사례 게시글) | posts 테이블 |
| 스터디 신청 | 오늘 결제 완료된 스터디 수강 신청 수 | orders 테이블 |
| 댓글 작성 | 오늘 작성된 댓글 수 | comments 테이블 |
| 상품 판매 | 오늘 결제 건수 및 매출 | payments 테이블 |
| 미처리 신고 | 미처리 상태 신고 건수 | reports 테이블 |

**수용 기준**:
- 통계 카드는 페이지 로드 시 즉시 표시 (캐시 활용, 1분 갱신)
- 각 카드 클릭 시 해당 관리 페이지로 이동
- 전일 대비 증감 표시 (예: +12, -3)

#### 3.1.2 빠른 작업

대시보드에서 자주 수행하는 작업에 바로 접근하는 버튼 그룹:

| 빠른 작업 | 이동 대상 | 사용 빈도 |
|-----------|----------|----------|
| 게시글 이동 | `/admin/posts` (이동 모드) | 매일 |
| 배너 수정 | `/admin/banners` | 주 2~3회 |
| 스터디 상태 변경 | `/admin/studies` | 기수 전환 시 집중 |
| 상품/쿠폰 관리 | `/admin/products` | 기수별 |
| 공지 작성 | `/write?type=case&category=notice` | 주 1~2회 |
| 텍스트 수정 | `/admin/texts` | 캠페인 시 |
| 기수 관리 | `/admin/cohorts` | 기수 전환 시 |
| 수료 판정 | `/admin/completion` | 기수 종료 후 |

**수용 기준**:
- 각 버튼 1클릭으로 해당 기능 화면에 도달
- 빠른 작업 목록은 사용 빈도 기반으로 정렬 (향후 커스터마이징 가능)

#### 3.1.3 최근 활동 피드

시간순으로 사이트 내 주요 활동을 실시간 표시:
- 게시글 작성/수정/삭제
- 회원 가입
- 스터디 신청/결제
- 댓글 작성
- 관리자 작업 로그 (배너 수정, 게시글 이동 등)

**수용 기준**:
- 최근 50건 표시, 무한 스크롤 또는 페이지네이션
- 각 활동 항목 클릭 시 해당 게시글/회원/스터디로 이동
- 활동 유형별 필터 가능

### 3.2 게시글 관리 (`/admin/posts`)

#### 3.2.1 게시글 목록

| 요소 | 설명 |
|------|------|
| 검색 | 제목, 작성자, 본문 키워드로 검색 |
| 카테고리 필터 | 6개 카테고리 드롭다운 (AI활용법, 프롬프트, 자동화/노코드, 개발/코딩, 비즈니스/마케팅, AI뉴스) |
| 상태 필터 | 공개 / 비공개 / 신고됨 |
| 콘텐츠 유형 필터 | 피드 포스트 / 사례 게시글 |
| 정렬 | 최신순 / 조회수순 / 댓글수순 |
| 테이블 컬럼 | 체크박스, 제목, 카테고리, 작성자, 작성일, 조회수, 상태, 작업 버튼 |

#### 3.2.2 카테고리 이동 기능 (P15 해결 핵심)

**현재**: 게시글 카테고리 이동 불가 -> 삭제 후 재게시 -> 웹훅 2회 발동

**개선**:
- 게시글 행의 "이동" 드롭다운에서 대상 카테고리 선택 -> 즉시 이동
- 일괄 이동: 체크박스로 여러 게시글 선택 -> "이동" 버튼 -> 대상 카테고리 선택
- 이동 시 웹훅 비발동 (이동은 `category_id` 업데이트만 수행, 새 게시글 생성 이벤트가 아님)
- 이동 이력 로그 기록 (누가, 언제, 어디서 어디로)

**수용 기준**:
- 단일 게시글: 행 내 드롭다운으로 1클릭 이동
- 일괄 이동: 선택 -> 이동 버튼 -> 카테고리 선택 -> 확인 (3클릭)
- 이동 후 원래 URL 유지 (`/posts/{slug}` 구조이므로 카테고리 변경과 무관)
- 이동 시 기존 댓글, 투표, 북마크 데이터 보존

#### 3.2.3 게시글 개별 작업

| 작업 | 설명 |
|------|------|
| 수정 | 관리자 편집 모드로 이동 |
| 이동 | 카테고리 변경 (3.2.2 참조) |
| 공개/비공개 | 게시글 가시성 토글 |
| 삭제 | 확인 다이얼로그 후 삭제 |
| 원본 보기 | 프론트 게시글 상세 페이지로 새 탭 이동 |

#### 3.2.4 일괄 작업

체크박스로 여러 게시글 선택 시 하단에 액션바 표시:

| 일괄 작업 | 설명 |
|-----------|------|
| 이동 | 선택된 게시글 일괄 카테고리 이동 |
| 공개/비공개 | 선택된 게시글 일괄 상태 변경 |
| 삭제 | 선택된 게시글 일괄 삭제 (확인 필수) |

**수용 기준**:
- 전체 선택 체크박스로 현재 페이지 전체 선택/해제
- 선택된 게시글 수가 액션바에 표시 ("N개 선택됨")
- 일괄 삭제는 반드시 확인 다이얼로그 표시

### 3.3 스터디 관리 (`/admin/studies`)

#### 3.3.1 스터디 목록

| 요소 | 설명 |
|------|------|
| 기수 필터 | 드롭다운으로 기수 선택 (최신 기수 기본 선택) |
| 상태 필터 | 작성중 / 모집중 / 진행중 / 완료 |
| 카드 표시 정보 | 스터디명, 스터디장, 인원(현재/정원), 상태, 최종제출 여부 |

#### 3.3.2 상태 변경

각 스터디 카드에서 상태를 드롭다운으로 즉시 변경:

```
작성중 → 모집중 → 진행중 → 완료
```

**수용 기준**:
- 드롭다운 변경 즉시 반영 (서버 업데이트)
- 상태 변경 시 확인 다이얼로그 표시 (되돌리기 어려운 변경이므로)
- 상태별 시각적 구분 (컬러 코딩: 작성중=muted, 모집중=primary, 진행중=accent, 완료=muted)

#### 3.3.3 최종제출 토글 (P13 해결 핵심)

**현재**: 게시글 진입 -> 수정 버튼 -> 긴 스크롤 -> 최종제출 탭 찾기 -> 토글 (6단계)

**개선**: 스터디 카드에서 **토글 스위치 1클릭**으로 최종제출 ON/OFF

**수용 기준**:
- 목록 화면에서 토글로 바로 조작 가능
- 토글 변경 즉시 서버 반영
- ON/OFF 상태가 시각적으로 명확히 구분
- 에어테이블 연동 상태 표시 (최종제출 ON -> 에어테이블 동기화 확인)

#### 3.3.4 참여자 현황

각 스터디의 상세보기에서 확인 가능한 정보:

| 정보 | 설명 |
|------|------|
| 수강생 목록 | 이름, 가입일, 결제 상태 |
| 과제 제출 현황 | 주차별 게시글 작성 여부 (체크 표시) |
| 출석 현황 | 줌 세션 참여 여부 |
| 수료 예측 | 현재 진도 기준 수료 가능 여부 표시 |

**수용 기준**:
- 스터디 카드에서 "상세보기" 클릭 시 참여자 현황 패널 표시
- 과제 미제출자를 한눈에 파악 가능 (미제출=빨간 표시)
- 수료 기준 충족 여부 자동 계산 (출석 + 과제 기반)

#### 3.3.5 랜딩페이지 연결

각 스터디 카드에서 "랜딩페이지" 버튼 클릭 시 프론트 스터디 상세 페이지(`/study/{slug}`)로 새 탭 이동.

**수용 기준**:
- 프론트 페이지와의 매핑이 명확 (P14 해결)
- 링크 아이콘(ExternalLink)으로 새 탭 이동임을 시각적으로 표시

#### 3.5.2 가격 자동 전환 설정

스터디 상품의 가격이 마감일 기반으로 자동 전환되는 규칙 설정:

```
슈퍼얼리버드가 (모집 시작 ~ D-14)
  → 얼리버드가 (D-14 ~ D-7)
    → 일반가 (D-7 ~ 모집 마감)
```

| 설정 항목 | 설명 |
|-----------|------|
| 슈퍼얼리버드 기간 | 시작일 ~ 종료일 |
| 얼리버드 기간 | 시작일 ~ 종료일 |
| 일반 기간 | 시작일 ~ 모집 마감일 |
| 각 단계 가격 | 원 단위 입력 |

**수용 기준**:
- 날짜 기반 자동 전환 (서버 스케줄러)
- 현재 적용 중인 가격 단계가 상품 목록에 시각적으로 표시
- 가격 전환 예정 일시 미리보기

#### 3.5.3 쿠폰 관리

| 요소 | 설명 |
|------|------|
| 쿠폰 목록 | 쿠폰 코드, 할인 유형(정액/정률), 사용 횟수/한도, 유효 기간, 상태 |
| 쿠폰 생성 | 코드(자동생성/수동입력), 할인 유형, 금액/비율, 적용 대상 상품, 사용 한도, 유효 기간 |
| 쿠폰 유형 | 스터디장 지인초대 쿠폰, 임직원 쿠폰, 프로모션 쿠폰 |

| 확정 | "수료 확정" 버튼으로 최종 반영 |

**수용 기준**:
- 자동 집계 결과를 미리보기로 확인 가능 (확정 전)
- 개별 수강생의 수료 상태를 수동으로 오버라이드 가능
- 확정 후에도 운영자가 개별 수정 가능 (이력 기록)
- 확정 시 수강생 대시보드에 수료 결과 자동 반영

#### 3.11.3 수료증 일괄 발급

| 항목 | 설명 |
|------|------|
| 발급 대상 | 수료 확정된 수강생 전체 또는 선택 |
| 수료증 생성 | 수료자 정보 + 스터디 정보 기반 PDF 자동 생성 |
| 우수활동자 표시 | 수료증에 우수활동자 마크 자동 포함 |
| 발급 이력 | 발급일, 대상자, 다운로드 여부 |

**수용 기준**:
- "일괄 발급" 버튼으로 수료자 전체 수료증 한 번에 생성
- 수료증 미리보기 제공 (발급 전 확인)
- 발급된 수료증은 수강생 대시보드에 자동 노출
- AI 이력서(프로필)에 수료 이력 자동 반영

#### 3.11.4 환급 관리

| 항목 | 설명 |
|------|------|
| 환급 대상자 추출 | 환급 요건(사례글 2회 이상 작성) 충족자 자동 필터링 |
| 환급 유형 | 버디 환급(100% - 100원), 일반 환급(조건별) |
| 환급 상태 | 대상 확인 → 환급 진행 중 → 환급 완료 |
| 환급 금액 | 결제 금액 기반 자동 계산 |

**수용 기준**:
- 기수별 환급 대상자 목록을 자동 추출
- 환급 상태를 건별로 관리 (대상 → 진행 중 → 완료)
- 환급 완료 시 수강생에게 알림 발송 (DM/이메일)
- 환급 내역 CSV 다운로드 (정산용)

### 3.14 과제 연동 권한 설정

스터디 핵심 자동화인 "과제 작성 → VOD 열람 권한 부여" 규칙을 어드민에서 관리한다.

#### 3.14.1 규칙 설정

| 항목 | 설명 |
|------|------|
| 기수별 ON/OFF | 기수 단위로 과제 연동 권한 규칙 활성/비활성 |
| 조건 | 이번 주 사례 게시글 N개 이상 작성 (기본값: 1개) |
| 범위 | 권한 획득 시 접근 가능한 VOD 범위 (기본값: 전체 15개 스터디) |
| 리셋 타이밍 | 주차 전환 시점 (기본값: 매주 월요일 00:00) |
| 1주차 정책 | 1주차는 과제 없음 — VOD 공개 여부 별도 설정 |

#### 3.14.2 예외 관리

| 항목 | 설명 |
|------|------|
| 개별 수강생 예외 | 특정 수강생에게 수동으로 VOD 접근 권한 부여/해제 |
| 스터디별 예외 | 특정 스터디는 과제 연동 없이 VOD 상시 공개 |
| 예외 이력 | 누가, 언제, 누구에게 예외를 적용했는지 기록 |

**수용 기준**:
- 과제 연동 규칙이 기수별로 독립 설정 가능
- 예외 처리 시 이력이 자동 기록
- 설정 변경은 운영자만 가능 (마케터/스터디장 접근 불가)
- 현재 과제 연동 현황 요약 표시 (N명 권한 획득 / N명 미획득)

#### 3.16.1 뱃지 정의

| 유형 | 조건 | 예시 |

- 뱃지 비활성화 시 이미 부여된 뱃지는 유지 (신규 부여만 중단)

## 4. 유저 플로우

### 4.1 운영자의 일과 (대표 시나리오)

```
[아침 루틴]
  로그인 → 프로필 드롭다운 → "관리자 대시보드"
       │
       ▼
  대시보드 (/admin)
       │
       ├── 오늘의 요약 확인 (신규가입 12명, 게시글 34개, 스터디 신청 5건)
       │
       ├── 최근 활동 피드 스캔
       │   └── "홍길동님 게시글을 잘못된 카테고리에 작성" 발견
       │        │
       │        ▼
       │   빠른 작업 → "게시글 이동" 클릭
       │        │
       │        ▼
       │   게시글 관리 (/admin/posts)
       │        ├── 해당 게시글 검색
       │        ├── 행 내 "이동" 드롭다운 → 올바른 카테고리 선택
       │        └── 완료 (1분 이내, 웹훅 미발동)
       │
       └── 사이드바 → "스터디" 클릭
              │
              ▼
         스터디 관리 (/admin/studies)
              ├── 기수 드롭다운: "21기" 선택
              ├── 각 스터디 카드에서 최종제출 토글 확인
              ├── 미제출 스터디장에게 "상세보기" → 참여자 현황 확인
              └── 상태 "작성중" → "모집중" 변경 (기수 전환 시)
```

### 4.2 마케터의 배너 수정 시나리오

```
[캠페인 시작]
  대시보드 → 빠른 작업 → "배너 수정"
       │
       ▼
  배너 관리 (/admin/banners)
       │
       ├── "홈 히어로 배너" 탭 선택
       ├── "새 배너" 클릭
       │   ├── 이미지 업로드
       │   ├── 링크 URL 입력
       │   ├── 노출 기간: 3/1 ~ 3/15 설정
       │   └── 저장
       │
       ├── 드래그앤드롭으로 새 배너를 1순위로 이동
       ├── "미리보기" 클릭 → 홈에서의 실제 모습 확인
       └── 완료
```

### 4.3 마케터의 문구 수정 시나리오

```
[문구 교체]
  대시보드 → 빠른 작업 → "텍스트 수정"

       └── "발행" 클릭 → 즉시 프론트 반영
```

### 4.4 스터디장의 관리 시나리오

## 5. IA -- 어드민 전용 정보구조

### 5.1 네비게이션 구조

```
관리자 대시보드 (/admin)
├── 대시보드        (/admin)             ← 기본 랜딩
├── 게시글          (/admin/posts)       ← 매일 사용
├── 스터디          (/admin/studies)     ← 매일 사용
├── 기수            (/admin/cohorts)     ← 기수 전환 시 집중  ← NEW
├── 배너            (/admin/banners)     ← 주 2~3회
├── 상품/쿠폰       (/admin/products)    ← 기수별
├── 결제/주문       (/admin/orders)      ← 필요 시           ← NEW
├── 수료/환급       (/admin/completion)  ← 기수 종료 후      ← NEW
├── 세션            (/admin/sessions)    ← 기수별             ← NEW
├── 회원            (/admin/users)       ← 필요 시
├── 신고/모더레이션  (/admin/moderation)  ← 필요 시           ← NEW
```

### 5.2 설계 원칙

| 원칙 | 설명 | 근거 |
|------|------|------|
| **Flat 구조** | 모든 메뉴가 사이드바 1단계에 노출. 중첩 없음 | P13: 7단계 → 1단계 |
| **1~2클릭 도달** | 모든 핵심 작업에 2클릭 이내 접근 | 신연권: "여러 번 헤맨다" |

| **인라인 조작** | 목록에서 바로 토글/변경 (별도 페이지 이동 최소화) | P13: 스크롤-편집-저장 반복 해소 |
| **역할 기반 노출** | 마케터/스터디장은 권한 범위의 메뉴만 표시 | 불필요한 메뉴 노이즈 제거 |

### 7.3 기술 참고 사항

- **사이드바**: 현재 각 페이지에 사이드바가 중복 정의되어 있음. 공통 레이아웃 컴포넌트로 추출 권장 (`app/admin/layout.tsx`)
- **데이터**: 현재 정적 mock 데이터. Supabase 연동 시 Server Component로 데이터 페칭, Client Component는 상호작용만 담당
- **상태 관리**: 토글, 드롭다운 등 인라인 조작은 `use client`로 구현. 서버 반영은 Server Action 또는 API Route 활용
- **권한 분기**: Supabase auth의 custom claims 또는 profiles 테이블의 role 필드 기반으로 사이드바 메뉴 조건부 렌더링
