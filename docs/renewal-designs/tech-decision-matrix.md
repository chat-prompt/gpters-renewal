## 1. Executive Summary (한 페이지 요약)

### 왜 리뉴얼하는가?

현재 GPTers 시스템은 **8개 이상의 서비스에 데이터가 흩어져** 있어, AI도 사람도 효율적으로 운영할 수 없습니다.

```
현재 상태 (데이터 분산):
  PostgreSQL (Neon)  → 회원, 결제, 수강
  Airtable           → 기수 관리, 운영 데이터 (9개 테이블)
  Bettermode         → 커뮤니티 게시글, 댓글
  Upstash Redis      → 세션, 캐시
  Cloudflare R2      → 파일 저장
  Cloudflare Workers → 자동화 (8개)
  KakaoTalk          → 공지, 리마인더
  Google Calendar    → 일정 관리

문제:
  - 회원 데이터 조회 시 PostgreSQL + Airtable + Bettermode 3곳을 다 봐야 함
  - AI가 GPTers 시스템을 이해하려면 8개 서비스 API를 모두 알아야 함
  - 운영진이 수동으로 데이터를 복사/붙여넣기 하는 작업이 매주 반복
  - 장애 시 어느 서비스가 문제인지 파악하기 어려움
```

### 해결 방향: AWS 올인 자체 구축

**모든 데이터와 서비스를 AWS 한 곳에 통합하여 자체 구축합니다.**

```
목표 상태 (AWS 통합):
  AWS RDS PostgreSQL → 모든 데이터 (회원, 결제, 커뮤니티, 스터디, LMS)
  AWS S3             → 모든 파일 (이미지, PDF, VOD 썸네일)
  AWS ElastiCache    → 세션, 캐시
  자체 인증 시스템    → 로그인, 권한 관리
  Next.js API Routes → 모든 비즈니스 로직
  Vercel             → 프론트엔드 호스팅

효과:
  - 하나의 DB에서 모든 데이터 조회/분석 가능
  - AI가 하나의 시스템만 이해하면 됨 (AI Native 운영)
  - 운영 자동화 확대 (수동 작업 제거)
  - 장애 원인 빠르게 파악 (한 곳에서 모니터링)
```

### 확정 사항

| 항목 | 결정 | 이유 |
|------|------|------|
| 전략 방향 | **AWS 올인 자체 구축** | 데이터 통합, AI Native 운영, Vendor Lock-in 탈피 |
| DB 호스팅 | **AWS RDS PostgreSQL** | 서울 리전, 완전 제어권, 장기 확장성 |

| 데이터 접근 제어 | **Application-level 인가** | 복잡한 비즈니스 로직 표현에 최적, 디버깅 용이 |

## 2. 왜 "레거시 유지"가 아니라 "자체 구축"인가?

### 2.1 현재 문제: 서비스 파편화

GPTers가 리뉴얼하는 핵심 이유는 **너무 많은 서비스에 데이터가 흩어져 있어서** AI도 사람도 효율적으로 운영할 수 없기 때문입니다.

| 현재 사용 서비스 | 역할 | 문제점 |
|-----------------|------|--------|
| PostgreSQL (Neon) | 회원, 결제, 수강 DB | 싱가포르 리전 (한국 느림) |
| Airtable | 기수 관리, 운영 데이터 | 9개 테이블에 핵심 운영 데이터. API 제한 있음. 수동 작업 많음 |
| Bettermode | 커뮤니티 (게시글, 댓글) | 커스텀 불가. Webhook으로 DB 동기화 필요. 데이터 소유권 불확실 |
| Upstash Redis | 세션, 캐시 | Vercel 전용. AWS와 별도 네트워크 |
| Cloudflare R2 | 파일 저장 | AWS와 별도 클라우드. 8개 Workers 관리 부담 |
| Cloudflare Workers | 리다이렉트, 캐시 (8개) | Cloudflare 종속. 디버깅 어려움 |
| KakaoTalk | 공지, 리마인더 | 자동화 불가. 수동 발송 |
| Google Calendar | 일정 | 시스템 미연동 |

### 2.2 "안정 우선" 전략의 문제

기존 v1.0 문서에서 권장했던 "안정 우선 전략" (Prisma 유지, R2 유지, Upstash 유지)은 **리뉴얼의 근본 목적과 충돌**합니다.

| "안정 우선" 접근 | 문제 |
|-----------------|------|
| Prisma 유지 | 레거시 코드 구조를 그대로 가져감. 신규 구축의 이점 없음 |
| Cloudflare R2 유지 | AWS와 다른 클라우드에 스토리지. 네트워크 비용 발생. Workers 유지보수 |
| Upstash Redis 유지 | AWS VPC 외부. 네트워크 지연. AWS 내부 통합 불가 |
| Better Auth 권장 | 2024년 출시 신생 라이브러리. 자체 JWT가 오히려 안정적 |

### 2.3 "AWS 올인 자체 구축" 전략의 장점

| 관점 | 장점 |
|------|------|
| **AI Native 운영** | 모든 데이터가 하나의 DB에 있어 AI가 전체 시스템을 이해하고 자동화 가능 |
| **운영 효율** | Airtable 수동 작업 제거. 어드민 대시보드에서 모든 운영 가능 |
| **데이터 통합** | 회원-결제-수강-커뮤니티-LMS 데이터가 한 곳. 교차 분석 가능 |
| **비용 예측** | AWS 한 곳의 비용만 관리. 여러 서비스 청구서 통합 |
| **보안 통합** | AWS VPC 내부에서 모든 서비스 간 통신. 외부 노출 최소화 |
| **확장성** | Read Replica, Multi-AZ, Auto Scaling 등 AWS 인프라 활용 |
| **장기 전략** | 자체 기술 역량 축적. 특정 서비스에 종속되지 않는 독립 아키텍처 |

## 3. 확정 사항 상세

### 3.1 DB: AWS RDS PostgreSQL

| 항목 | 기존 (Neon) | 변경 (AWS RDS) |
|------|------------|---------------|
| 쉬운 설명 | 서버리스 DB (쓴 만큼 과금) | AWS에서 관리하는 전용 DB 서버 |
| 서울 리전 | 미지원 (싱가포르) | **지원** (ap-northeast-2) |
| 응답 속도 | 80~150ms (Cold Start) | **10~30ms** (상시 가동) |
| 자동 백업 | 7일 | **35일** (설정 가능) |
| Read Replica | 미지원 | **지원** (읽기 부하 분산) |
| Multi-AZ | 미지원 | **지원** (고가용성) |
| 월 비용 | ~$25 (Pro) | ~$35~60 (db.t4g.medium) |

**왜 AWS RDS인가?**

1. **서울 리전**: 한국 사용자 체감 속도 대폭 개선 (DB 응답 최대 5배 빠름)
2. **AWS 생태계**: S3, ElastiCache, CloudFront와 VPC 내부 통신 (빠르고 안전)
3. **완전한 제어**: 인스턴스 타입, 백업 주기, 보안 그룹을 직접 설정
4. **장기 확장**: 트래픽 증가 시 Read Replica 추가, 인스턴스 업그레이드 용이

### 3.2 스토리지: AWS S3 + CloudFront

| 항목 | 기존 (Cloudflare R2) | 변경 (AWS S3 + CloudFront) |
|------|--------------------|-----------------------|
| 쉬운 설명 | Cloudflare 파일 저장소 | AWS 파일 저장소 + 글로벌 CDN |
| AWS 통합 | 별도 클라우드 | **VPC 내부** (DB와 같은 네트워크) |
| CDN | Cloudflare 엣지 | **CloudFront** (250+ 엣지 로케이션) |
| Workers 의존 | 8개 Workers 필요 | **불필요** (Lambda@Edge 또는 API Routes) |
| 이그레스 비용 | 무료 | $0.09/GB (CloudFront 할인 적용 시 저렴) |
| 저장 비용 | $0.015/GB/월 | $0.023/GB/월 |
| 현재 규모 월 비용 | ~$2 | ~$5~10 |

**왜 S3인가?**

1. **AWS 통합**: RDS, ElastiCache와 같은 VPC. 네트워크 비용 없음
2. **Workers 제거**: 8개 Cloudflare Workers 유지보수 부담 해소
3. **CloudFront 통합**: 이미지 최적화, 캐싱, 한국 사용자 빠른 응답
4. **Lambda@Edge**: 이미지 리사이징 등을 AWS 내부에서 처리
5. **비용**: 현재 규모(~10GB)에서 월 $5~10. R2 대비 약간 비싸지만 통합 관리의 가치

**기존 파일 이전**: R2 → S3로 일괄 복사 후 URL 리다이렉트 설정. 사용자 영향 없음.

### 3.3 캐시/세션: AWS ElastiCache (Redis)

| 항목 | 기존 (Upstash Redis) | 변경 (AWS ElastiCache) |
|------|--------------------|-----------------------|
| 쉬운 설명 | 서버리스 고속 메모장 (HTTP API) | AWS 전용 고속 메모장 (TCP 직접 연결) |
| 연결 방식 | HTTP REST API | **TCP 직접** (VPC 내부) |
| 응답 속도 | 1~5ms (HTTP 오버헤드) | **<1ms** (TCP 직접) |
| AWS 통합 | 별도 서비스 | **VPC 내부** (RDS와 같은 네트워크) |
| Vercel 연동 | 네이티브 SDK | VPC 연결 필요 (Vercel Secure Compute) |
| Rate Limiting | @upstash/ratelimit 내장 | 직접 구현 (간단) |
| 월 비용 | ~$3~5 | ~$15~30 (cache.t4g.micro) |

**왜 ElastiCache인가?**

1. **VPC 내부 통신**: RDS, S3와 같은 네트워크. 보안과 속도 모두 확보
2. **<1ms 응답**: HTTP REST 오버헤드 없이 TCP 직접 연결
3. **AWS 통합 모니터링**: CloudWatch에서 Redis 메트릭 통합 관리
4. **보안**: VPC Security Group으로 접근 제어. 외부 노출 없음

**Vercel 연동 방법**: Vercel Secure Compute (Pro 플랜 포함) 또는 API Routes에서 VPC 내부 연결

### 3.4 데이터 접근 제어: Application-level 인가

| 역할 | 쉬운 설명 | 접근 범위 |
|------|----------|----------|
| guest (비회원) | 로그인하지 않은 방문자 | 공개 게시글, 스터디 목록 조회 |
| member (회원) | 가입한 일반 회원 | + 게시글 작성, 댓글, 투표 |
| student (수강생) | 스터디에 결제/등록한 회원 | + 과제 제출, VOD 접근, LMS |
| leader (스터디장) | 스터디를 운영하는 사람 | + 스터디 관리, 수강생 현황 |
| admin (운영자) | GPTers 운영팀 | 전체 접근 |

**구현 방식**:

```
사용자가 VOD를 보려고 할 때:

1. 서버가 "이 사용자 로그인했나?" 확인 → 안 했으면 로그인 페이지로
2. "이 사용자가 이 스터디에 수강 등록했나?" 확인 → 안 했으면 차단
3. "이번 주 과제를 제출했나?" 확인 → 안 했으면 "과제를 먼저 제출하세요" 표시
4. 모든 조건 통과 → VOD 재생
```

### 3.5 Airtable 이관 → AWS RDS

현재 Airtable에 있는 9개 테이블을 모두 AWS RDS PostgreSQL로 이관합니다.

| Airtable 테이블 | 역할 | RDS 이관 테이블 |
|-----------------|------|----------------|
| Studies | 스터디 정보 | `studies` |
| Cohorts | 기수 정보 | `cohorts` |
| Enrollments | 수강 등록 | `enrollments` |
| Assignments | 과제 현황 | `assignments` |
| Attendance | 출석 | `attendance` |
| VOD | 다시보기 | `vod_recordings` |
| Leaders | 스터디장 | `study_leaders` |
| Schedule | 일정 | `study_schedules` |
| Certificates | 수료증 | `certificates` |

**이관 후 효과**:
- Airtable 의존 완전 제거 (API 호출 제한 해소)
- 운영 데이터가 회원/결제 데이터와 같은 DB → JOIN 쿼리로 즉시 분석
- AI가 전체 데이터를 한 번에 이해 가능

### 3.6 Bettermode 제거 → 자체 커뮤니티

| 항목 | 기존 (Bettermode) | 변경 (자체 구축) |
|------|-------------------|----------------|
| 게시글/댓글 | Bettermode API + Webhook 동기화 | AWS RDS `posts`, `comments` 테이블 |
| 데이터 소유권 | Bettermode가 SoR (원본) | **자체 DB가 SoR** |
| 커스텀 기능 | 제한적 (Bettermode 제약) | **무제한** (투표, 태그, 필터 자유) |
| 동기화 비용 | Webhook 핸들러 유지보수 | **불필요** (단일 시스템) |

## 4. 의사결정 필요 항목 (2개)

### 4.1 인증(로그인) 시스템

> **쉬운 설명**: 사용자가 카카오/네이버/이메일로 로그인할 때 사용하는 기술.
> "로그인 버튼을 누르면 뒤에서 누가 처리하나?"에 해당합니다.

#### 방안 비교표

| 비교 항목 | A. 자체 JWT + OAuth (권장) | B. AWS Cognito | C. NextAuth.js (Auth.js) |
|----------|------------------------|---------------|------------------------|
| **쉬운 설명** | 로그인 시스템을 처음부터 직접 만듦. 완전한 제어 | AWS가 제공하는 로그인 서비스. AWS 생태계 통합 | Next.js 생태계의 로그인 도구. 커뮤니티 활발 |
| **카카오 로그인** | 직접 구현 (카카오 API 연동) | OIDC 커스텀 프로바이더로 연동 | 공식 프로바이더 지원 |
| **네이버 로그인** | 직접 구현 (네이버 API 연동) | OIDC 커스텀 프로바이더로 연동 | 커뮤니티 프로바이더 |
| **이메일 로그인** | 직접 구현 | 내장 지원 | 내장 지원 (Magic Link) |
| **AWS 통합** | API Routes에서 RDS/ElastiCache 직접 사용 | **네이티브** (IAM, Lambda Trigger) | Adapter로 연동 |
| **세션 관리** | JWT + ElastiCache Redis | Cognito 토큰 (자동 관리) | JWT or DB 세션 |
| **Vendor 종속** | **없음** (표준 기술만 사용) | AWS Cognito 종속 | Auth.js 의존 |
| **데이터 소유** | **RDS에 직접 저장** | Cognito User Pool (별도 저장소) | RDS에 저장 (Adapter) |
| **커스텀 자유도** | **최대** | 제한적 (Cognito 규칙 따라야 함) | 중간 |
| **개발 기간** | 7~10일 | 5~7일 | 3~5일 |
| **월 추가 비용** | $0 | $0 (MAU 50K까지 무료) | $0 |
| **기존 코드 재활용** | 30% (레거시 OAuth 로직 참조) | 20% (전면 재작성) | 40~50% |

#### 권장안: A. 자체 JWT + OAuth

**이유**:

1. **완전한 데이터 소유**: 사용자 데이터가 AWS RDS에 직접 저장. 별도 서비스(Cognito User Pool)에 분산되지 않음
2. **AI Native**: 인증 데이터도 하나의 DB에 있어 AI가 전체 사용자 여정을 이해 가능
3. **무제한 커스텀**: 과제 연동 권한, 기수별 접근 제어 등 GPTers 고유 로직을 자유롭게 구현
4. **Vendor 종속 없음**: jose (JWT), bcrypt (비밀번호) 등 표준 라이브러리만 사용
5. **장기 전략**: 자체 인증 역량 확보. 어떤 프레임워크/클라우드로든 이식 가능

**구현 핵심**:

```
로그인 흐름 (카카오 예시):

1. 사용자가 "카카오로 로그인" 클릭
2. 카카오 로그인 페이지로 이동 (OAuth 표준 흐름)
3. 카카오에서 인증 성공 → 우리 서버로 코드 전달
4. 서버가 코드로 카카오 사용자 정보 조회
5. 우리 DB(RDS)에서 사용자 찾거나 새로 생성
6. JWT 토큰 발급 → 쿠키에 저장
7. 이후 모든 요청에 JWT로 사용자 확인

보안:
- Access Token: 15분 (짧게 → 탈취 시 피해 최소화)
- Refresh Token: 7일 (ElastiCache에 저장, 로그아웃 시 즉시 무효화)
- 쿠키: httpOnly + Secure + SameSite=Lax
```

**왜 다른 방안은 비추천인가?**

| 방안 | 비추천 이유 |
|------|-----------|
| B. AWS Cognito | 사용자 데이터가 Cognito User Pool에 별도 저장 → 데이터 통합 목표와 충돌. 카카오/네이버 OAuth 연동이 복잡 (OIDC 커스텀 필요) |
| C. NextAuth.js | Auth.js 라이브러리에 종속. v5에서 Breaking Change 많음. Credentials Provider 공식 비권장. 자체 구축보다 제어력 낮음 |

**위험 요소**:
- 직접 구현이므로 보안 검증에 주의 필요 (OWASP Top 10 체크리스트 적용)
- 카카오/네이버 OAuth API 변경 시 직접 대응 필요

**기존 사용자 영향**:
- 마이그레이션 후 **1회 재로그인 필요** (자동 리다이렉트)
- 카카오 계정 연동은 그대로 유지 (OAuth Provider ID 기반 매칭)

### 4.2 ORM (데이터베이스 접근 도구)

> **쉬운 설명**: 개발자가 코드에서 DB의 데이터를 읽고 쓸 때 사용하는 도구.
> "회원 정보를 가져와줘"라고 코드로 명령할 때, 그 명령을 DB가 이해하는 언어(SQL)로 번역해주는 통역사입니다.

#### 방안 비교표

| 비교 항목 | A. Drizzle ORM (권장) | B. Prisma | C. 순수 SQL (no ORM) |
|----------|---------------------|-----------|---------------------|
| **쉬운 설명** | 가볍고 빠른 최신 통역사. SQL에 가까워 성능 예측 쉬움 | 가장 많이 쓰이는 통역사. 레거시에서 사용 중 | 통역사 없이 직접 DB 언어로 소통. 완전 제어 |
| **타입 안전성** | 강함 (자동 추론) | 매우 강함 (자동 생성) | 없음 (직접 정의) |
| **SQL 친화도** | **매우 높음** (SQL과 1:1 매핑) | 낮음 (Prisma 자체 문법) | 100% SQL |
| **Edge Runtime** | **완벽 지원** | 제한적 (Accelerate 필요) | 완벽 지원 |
| **번들 사이즈** | **작음** (~50KB) | 큼 (~2MB, 엔진 포함) | 최소 |
| **성능** | **매우 좋음** | 좋음 | 최상 |
| **마이그레이션 도구** | drizzle-kit (push/migrate) | prisma migrate | 직접 관리 |
| **RDS 연동** | 완벽 | 완벽 | 완벽 |
| **신규 프로젝트 적합도** | **최적** (새로 시작) | 레거시 유지에 적합 | 소규모 프로젝트에 적합 |
| **커뮤니티** | 빠르게 성장 (2023~) | 매우 성숙 (2019~) | - |
| **월 추가 비용** | $0 | $0 | $0 |

#### 권장안: A. Drizzle ORM

**이유**:

1. **신규 구축에 최적**: 레거시 Prisma 스키마를 가져올 필요 없이, 통합된 새 스키마를 처음부터 설계
2. **SQL 친화적**: 복잡한 쿼리(과제 연동 권한, 기수별 통계 등)를 SQL에 가까운 문법으로 직관적으로 작성
3. **경량**: Prisma 엔진(~2MB) 없이 가볍게 동작. Vercel 서버리스 환경에서 Cold Start 빠름
4. **Edge Runtime 완벽 지원**: 향후 Edge 환경 활용 시 제약 없음
5. **성능**: SQL에 직접 매핑되어 예측 가능한 쿼리 생성. Prisma 대비 10~30% 빠른 응답

**왜 다른 방안은 비추천인가?**

| 방안 | 비추천 이유 |
|------|-----------|
| B. Prisma | 레거시 구조를 신규 프로젝트에 가져오는 것은 기술 부채. 무거운 엔진. Edge 제약 |
| C. 순수 SQL | 타입 안전성 없음. 개발 생산성 낮음. GPTers 31개 테이블 규모에서는 ORM이 필수 |

**레거시 Prisma 스키마 활용 방법**:
- 기존 24개 Prisma 모델을 **참조**하여 Drizzle 스키마를 새로 설계
- 테이블 구조는 유사하되, Airtable 9개 테이블 + Bettermode 데이터를 통합한 확장 스키마
- 마이그레이션 스크립트에서 기존 Prisma 타입 활용하여 데이터 변환

## 7. 마이그레이션 영향도

### 7.1 사용자 영향 요약

| 영향 항목 | 영향도 | 사용자 행동 필요 | 설명 |
|----------|--------|---------------|------|
| 재가입 | **없음** | 불필요 | 기존 계정 DB 마이그레이션으로 자동 이전 |
| 재로그인 | **1회** | 로그인 버튼 클릭 | 인증 시스템 변경으로 기존 세션 만료 |
| 비밀번호 재설정 | **없음** | 불필요 | 카카오 OAuth 기반이므로 비밀번호 개념 없음 |
| 게시글/댓글 | **보존** | 없음 | Bettermode → 자체 DB로 전체 이관 |
| 결제 내역 | **보존** | 없음 | Neon → RDS로 전체 이관 |
| 수강 이력 | **보존** | 없음 | Neon + Airtable → RDS로 통합 이관 |
| 프로필 사진 | **보존** | 없음 | R2 → S3로 파일 이전 + URL 리다이렉트 |

### 7.2 데이터 이관 범위

```
Phase 1: PostgreSQL 이관 (Neon → RDS)
  - 24개 테이블 데이터 (pg_dump/pg_restore)
  - 사용자, 결제, 수강, 게시글 캐시 등
  - 검증: 레코드 수 일치, FK 무결성, 금액 합계

Phase 2: Airtable 이관 (9개 테이블 → RDS)
  - LMS 운영 데이터: 기수, 과제, 출석, VOD, 수료증 등
  - Airtable API로 데이터 추출 → RDS INSERT
  - 검증: 필드 매핑, 타입 변환, 관계 무결성

Phase 3: Bettermode 이관 (커뮤니티 → RDS)
  - 게시글, 댓글 데이터 (PostgreSQL CommunityPost에 캐시 있음)
  - 누락 데이터는 Bettermode API로 보완
  - 검증: 게시글 수 일치, 작성자 매핑

Phase 4: 파일 이관 (R2 → S3)
  - 프로필 사진, 게시글 이미지, 수료증 PDF
  - rclone 또는 AWS DataSync로 일괄 복사
  - 기존 URL → 새 URL 리다이렉트 설정
```

## Appendix A: AI Native 운영이란?

"AI Native"란 **AI가 GPTers 시스템을 완전히 이해하고 운영을 자동화**할 수 있는 상태를 말합니다.

### 현재 (AI가 할 수 없는 것)

```
- "이번 기수 수강생 중 과제 미제출자는?" → Airtable + PostgreSQL 2곳을 봐야 답 가능
- "게시글 작성 수 기준 상위 10명은?" → Bettermode API + PostgreSQL 2곳 필요
- "환불 후 커뮤니티 접근 차단" → PostgreSQL 환불 처리 + Bettermode 스페이스 제거 2단계
- "수강 현황 리포트 생성" → Airtable에서 수동 추출 후 가공
```

### 목표 (AI가 할 수 있는 것)

```
- "이번 기수 수강생 중 과제 미제출자는?" → 단일 SQL 쿼리로 즉시 답변
- "게시글 작성 수 기준 상위 10명은?" → 단일 SQL 쿼리
- "환불 후 커뮤니티 접근 차단" → 하나의 API 호출로 완료
- "수강 현황 리포트 생성" → 자동 생성 + 슬랙 발송
- "다음 기수 가격 자동 전환 설정" → 어드민 대시보드에서 클릭 1번
```

## Appendix B: 용어 해설

비개발자를 위한 주요 용어 설명입니다.

| 용어 | 쉬운 설명 |
|------|----------|
| **AWS RDS** | Amazon이 운영하는 데이터베이스 호스팅 서비스. "DB 서버를 빌려 쓰는 것" |
| **PostgreSQL** | 무료 오픈소스 데이터베이스. 세계에서 가장 많이 쓰이는 DB 중 하나 |
| **VPC** | Virtual Private Cloud. AWS 안에 만드는 전용 네트워크. "가상의 사설 네트워크" |
| **ElastiCache** | AWS의 고속 임시 저장소 서비스. Redis를 AWS에서 관리해주는 것 |
| **S3** | AWS의 파일 저장 서비스. "무한대 용량의 클라우드 파일 서버" |
| **CloudFront** | AWS의 CDN. 전 세계에 분산된 서버로 파일을 빠르게 전달 |
| **ORM** | 코드와 DB 사이의 통역사. 개발자가 SQL을 직접 안 써도 되게 해주는 도구 |
| **OAuth** | "카카오로 로그인" 같은 소셜 로그인 기술. 비밀번호 없이 다른 서비스 계정으로 로그인 |
| **JWT** | JSON Web Token. 로그인 정보를 암호화된 토큰으로 주고받는 방식 |
| **세션** | "이 사용자가 로그인 중이다"를 서버가 기억하는 방법 |
| **캐시** | 자주 쓰는 데이터를 빠른 저장소에 임시로 보관하여 속도를 높이는 기술 |
| **Redis** | 초고속 임시 저장소. "순간 메모장"처럼 빠르게 읽고 쓸 수 있는 DB |
| **Edge Runtime** | 사용자와 가까운 서버에서 코드를 실행하여 응답 속도를 높이는 기술 |
| **Vendor Lock-in** | 특정 서비스에 종속되어 다른 서비스로 옮기기 어려운 상태 |
| **AI Native** | AI가 시스템을 완전히 이해하고 운영 자동화가 가능한 상태 |
| **Multi-AZ** | 여러 물리적 데이터센터에 DB를 복제하여 장애에도 서비스 지속하는 기술 |
| **CDN** | Content Delivery Network. 전 세계에 분산된 서버로 파일을 빠르게 전달하는 기술 |
| **tRPC** | TypeScript 기반 API 통신 도구. 프론트엔드와 백엔드가 타입을 공유 |
| **Rate Limiting** | API 호출 횟수를 제한하여 서버 남용을 방지하는 기술 |
